## 什么是agent skill？

它其实就是一个大模型随时可以翻阅的说明文档。

例如：

智能客服skill

```
回答时请注意：
1. 必须先安抚用户
2. 不得随意承诺
```

会议总结skill

```
请将会议内容总结为如下几点
1. 参会人员
2. 议题
3. 决定
```

有了这些skill你就不用每次对话都去重复粘贴那一长串要求了，大模型自己翻翻这个说明文档就知道要怎么干活了

````
# 会议总结助手

此技能帮助用户总结会议录音并从音频文件中提取关键信息。

## 概述

会议总结助手处理会议录音，生成结构化摘要、提取待办事项、识别关键决策，并创建有条理的会议笔记。

## 适用场景

当用户的请求涉及以下内容时，此技能会激活：
- 总结会议录音
- 将会议音频转录为文字
- 从音频中提取会议笔记
- 创建会议摘要
- 处理会议录音（MP3、WAV、M4A 等）
- 从会议中生成待办事项
- 识别会议讨论中的关键决策

## 支持的功能

### 音频处理
- 使用语音识别将会议音频转录为文字
- 支持常见音频格式（MP3、WAV、M4A、AAC、FLAC）
- 尽可能识别多位发言人

### 摘要生成
- 生成简洁的会议摘要
- 提取主要议题和讨论要点
- 识别会议中做出的关键决策
- 列出待办事项及负责人（如有提及）
- 突出重要的日期或截止日期

### 输出格式
- 结构化文本摘要
- 要点列表
- 带负责人的待办事项列表
- 决策记录

## 工作流程

1. **音频输入**：接收会议录音文件路径或 URL
2. **转录**：将语音转换为文字（使用可用的工具/接口）
3. **分析**：处理文字稿，提取：
   - 讨论的主要议题
   - 做出的关键决策
   - 待办事项及负责人
   - 重要的日期/截止日期
   - 提出的问题
4. **生成摘要**：按要求的格式创建结构化摘要

## 最佳实践

- 询问用户偏好的输出格式（详细版或简洁版）
- 如果一个录音包含多个会议，请请求澄清
- 确认是否需要在摘要中特别强调特定主题
- 询问是否需要单独提取待办事项

3. 返回上传结果给用户

### 触发流程
1. **触发检测**：识别用户是否要求上传/同步/发送
2. **内容提取**：获取当前会议总结的完整内容
3. **执行上传**：运行 `upload_summary.sh` 脚本
4. **结果反馈**：告知用户上传成功或失败
````

```
#!/bin/bash

# 会议总结上传脚本
# 用法: ./upload_summary.sh <总结内容文件路径或直接输入内容>
# 或: cat <总结文件> | ./upload_summary.sh

# ==================== 配置区域 ====================
# 服务器上传地址（请根据实际情况修改）
SERVER_URL="https://your-server.com/api/upload"

# API 密钥（可选）
API_KEY=""
# ==================== 配置结束 ====================

# 颜色定义
RED='\033[0;31m'
GREEN='\033[0;32m'
YELLOW='\033[1;33m'
NC='\033[0m' # No Color

# 显示帮助
show_help() {
    echo "会议总结上传脚本"
    echo ""
    echo "用法:"
    echo "  $0 <内容>              直接上传内容"
    echo "  $0 -f <文件路径>       从文件上传内容"
    echo "  $0 -h                  显示帮助"
    echo ""
    echo "示例:"
    echo "  $0 \"会议总结内容...\""
    echo "  $0 -f /path/to/summary.txt"
    echo "  cat summary.txt | $0"
}

# 检查依赖
check_dependencies() {
    if ! command -v curl &> /dev/null; then
        echo -e "${RED}错误: curl 未安装${NC}"
        exit 1
    fi
}

# 上传内容
upload_content() {
    local content="$1"

    if [ -z "$content" ]; then
        echo -e "${RED}错误: 上传内容不能为空${NC}"
        exit 1
    fi

    echo -e "${YELLOW}正在上传会议总结...${NC}"

    # 构建请求
    if [ -n "$API_KEY" ]; then
        response=$(curl -s -w "\n%{http_code}" -X POST "$SERVER_URL" \
            -H "Content-Type: application/json" \
            -H "Authorization: Bearer $API_KEY" \
            -d "{\"content\": \"$content\", \"timestamp\": \"$(date -u +%Y-%m-%dT%H:%M:%SZ)\"}")
    else
        response=$(curl -s -w "\n%{http_code}" -X POST "$SERVER_URL" \
            -H "Content-Type: application/json" \
            -d "{\"content\": \"$content\", \"timestamp\": \"$(date -u +%Y-%m-%dT%H:%M:%SZ)\"}")
    fi

    # 分离 HTTP 状态码和响应体
    http_code=$(echo "$response" | tail -n1)
    response_body=$(echo "$response" | sed '$d')

    if [ "$http_code" -eq 200 ] || [ "$http_code" -eq 201 ]; then
        echo -e "${GREEN}上传成功!${NC}"
        echo "服务器响应: $response_body"
    else
        echo -e "${RED}上传失败! HTTP 状态码: $http_code${NC}"
        echo "错误信息: $response_body"
        exit 1
    fi
}

# 主函数
main() {
    check_dependencies

    local content=""

    # 处理管道输入
    if [ ! -t 0 ]; then
        content=$(cat)
    fi

    # 解析命令行参数
    while [[ $# -gt 0 ]]; do
        case $1 in
            -h|--help)
                show_help
                exit 0
                ;;
            -f|--file)
                if [ -z "$2" ]; then
                    echo -e "${RED}错误: -f 参数需要文件路径${NC}"
                    exit 1
                fi
                if [ ! -f "$2" ]; then
                    echo -e "${RED}错误: 文件不存在: $2${NC}"
                    exit 1
                fi
                content=$(cat "$2")
                shift
                ;;
            *)
                if [ -z "$content" ]; then
                    content="$1"
                fi
                ;;
        esac
        shift
    done

    if [ -z "$content" ]; then
        show_help
        exit 1
    fi

    upload_content "$content"
}

main "$@"

```

## 高级用法

Reference: 读，会消耗token

Script：跑，只会被执行，不会消耗token

### Reference

当我们的会议总结助手越来越高级，我们希望它不仅仅是简单的复述，而是能够提供更有价值的补充说明，比如说，当会议决定要花钱的时候，它能直接在总结里标注是否符合财务合规，当涉及到合同时候，它能提示法务分险，这样大家在看会议总结的时候就不用去看规章制度，一眼就能看到这些关键的补充信息， 这就非常方便了，但是问题在于Skill能做这些事情的前提是，它要把相关的财务规定和法律条文，都写入到SKILL.md文件里面，这些文件可能会非常的长，都写入进去的话，SKILL.md可能会非常的臃肿，哪怕只是开个简单的早会，都会都要被迫加载一堆根本用不上的财务和法律废话，浪费模型中的资源。
那能不能做到按需中的按需呢，比如说只有当会议内容真的聊到了钱，Claude Code才会把财务规定加载给模型看，这就要用到Reference了。

特点：条件触发的，在上面的例子2里，只有当Claude Code读取完skill.md文件，判断出需要查账时，才会去加载这个文件。反过来说，如果是一个与钱无关的会议，那么这个财务文件就只会躺在硬盘里。

### Script

![image-20260227221255841](/Users/liuruyue/Library/Application Support/typora-user-images/image-20260227221255841.png)

这里是Claude Code申请去执行这个脚本文件，它并没有去读取这个文件 ，这就意味着哪怕你的脚本写了一万行代码，它消耗的模型上下文也几乎为0，Claude Code只关心脚本的运行方法和运行结果，对脚本的内容毫不关心。

## 原理

![image-20260227215605997](/Users/liuruyue/Library/Application Support/typora-user-images/image-20260227215605997.png)

Agent Skill的第一个核心机制，按需加载，虽然skill的名字和描述是始终对模型可见的，但具体的指令内容，只有在这个skill被选中之后，才会被加载进来给模型看，这样就节省了很多Token了。
