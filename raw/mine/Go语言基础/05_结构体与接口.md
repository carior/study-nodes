# Go 结构体与接口

> [!warning] 本章重点
> **接口（interface）**是前端开发者最陌生的概念，也是 Go 最强大的特性之一。务必弄懂"什么是接口"、"怎么实现接口"，不要跳过。

---

## 一、结构体（struct）

### 定义与创建

```go
// 定义结构体
type User struct {
    Name string
    Age  int
    Email string
}

// 创建（多种方式）
u1 := User{Name: "张三", Age: 25}
u2 := User{}                        // 零值：{"", 0, ""}
u3 := new(User)                     // 返回 *User 指针
```

### 字段标签（JSON 序列化）

```go
type User struct {
    Name  string `json:"name"`
    Age   int    `json:"age"`
    Email string `json:"-"`        // 忽略此字段
}
```

> [!tip] 和前端的关系
> 结构体的字段标签控制 JSON 序列化行为：
> - Go 字段 `Name` → JSON 键 `name`（蛇形/驼峰转换）
> - `json:"-"` 的字段不会出现在 JSON 中
> 打开 Postman 调接口时，响应的 JSON 键名就由这里控制。

### 嵌套结构体

```go
type Address struct {
    City    string
    Country string
}

type User struct {
    Name    string
    Address Address  // 嵌套
}

u := User{
    Name: "张三",
    Address: Address{
        City:    "北京",
        Country: "中国",
    },
}
```

## 二、方法（Method）

Go 的方法绑定到结构体上（类似其他语言的方法）：

```go
type User struct {
    Name string
    Age  int
}

// 值接收者：不修改原结构体
func (u User) Greet() string {
    return "你好，我是" + u.Name
}

// 指针接收者：可以修改原结构体
func (u *User) Birthday() {
    u.Age++
}
```

> [!tip] 什么时候用指针接收者？
> - 要修改结构体字段 → 用 `*User`
> - 结构体较大，想避免拷贝 → 用 `*User`
> - 不确定？先用指针接收者 `*User`

### 使用

```go
u := User{Name: "张三", Age: 25}
fmt.Println(u.Greet()) // 你好，我是张三

u.Birthday()
fmt.Println(u.Age)      // 26
```

## 三、接口（Interface）⭐

### 什么是接口？

接口定义了一组**方法签名**，但不实现它们。任何类型只要实现了这些方法，就"隐式地"实现了该接口。

```go
// 定义接口
type Speaker interface {
    Speak() string
}

// Dog 实现了 Speaker 接口
type Dog struct{}

func (d Dog) Speak() string {
    return "汪汪！"
}

// Cat 也实现了 Speaker 接口
type Cat struct{}

func (c Cat) Speak() string {
    return "喵喵！"
}
```

> [!example] 为什么要接口？
> 函数可以接收 `Speaker` 接口类型，兼容所有实现了它的类型：
> ```go
> func talk(s Speaker) {
>     fmt.Println(s.Speak())
> }
>
> talk(Dog{}) // 汪汪！
> talk(Cat{}) // 喵喵！
> ```
> 这就是**多态**！同一个函数，传入不同类型，执行不同行为。

### 接口的零值

```go
var s Speaker  // nil
// s.Speak() 会 panic！因为 s 没有实际值
```

### 接口的类型断言

从接口变量中提取具体类型：

```go
var s Speaker = Dog{}

if dog, ok := s.(Dog); ok {
    fmt.Println("是 Dog:", dog)
}
```

## 四、常见标准库接口

Go 标准库大量使用接口，提前了解它们可以快速读懂源码：

| 接口 | 方法 | 谁实现了 |
|------|------|---------|
| `error` | `Error() string` | 所有返回错误的函数 |
| `io.Reader` | `Read(p []byte) (n int, err error)` | 文件、网络连接等 |
| `io.Writer` | `Write(p []byte) (n int, err error)` | 文件、标准输出等 |
| `json.Marshaler` | `MarshalJSON() ([]byte, error)` | 自定义 JSON 序列化 |

### io.Reader 示例

```go
// 这个函数可以接收：文件、网络连接、字符串缓冲区...
func readAll(r io.Reader) ([]byte, error) {
    return io.ReadAll(r)
}

// 用法
file, _ := os.Open("test.txt")
data, _ := readAll(file)
```

> [!success] 接口的威力
> `io.Reader` / `io.Writer` 接口让 Go 的函数极度灵活：
> 同一个 `readAll` 函数，可以读文件、读网络、读内存——只要实现了接口就行。
> 这就是 Go 的**依赖倒置**：函数依赖接口，不依赖具体实现。

## 五、空接口（interface{}）

空接口没有任何方法，所有类型都实现了它：

```go
var anything interface{} = 42
anything = "hello"
anything = []int{1, 2, 3}

// 相当于前端的 any
```

> [!warning] 少用空接口
> 空接口绕过了类型检查，用多了会丢失 Go 的类型安全优势。
> 尽量用具体接口或泛型（Go 1.18+）。

## 六、接口与前端的类比

| 概念 | Go | 前端 JS |
|------|-----|---------|
| 数据容器 | `struct` | `class` / `object` |
| 方法绑定 | `(u *User) Method()` | `class User { method() {} }` |
| 接口定义 | `type Reader interface { Read() }` | —（JS 无直接对应）|
| 接口实现 | 隐式实现（不需要 implements）| — |
| 多态 | 函数接收接口类型 | `class implements Interface` |

## 七、实战：HTTP 接口中的结构体与接口

前后端交互时，最常见的使用场景：

```go
// 1. 请求结构体
type CreateUserRequest struct {
    Name  string `json:"name"`
    Age   int    `json:"age"`
}

// 2. 响应结构体
type UserResponse struct {
    ID    int    `json:"id"`
    Name  string `json:"name"`
    Age   int    `json:"age"`
}

// 3. 接口（定义业务行为）
type UserService interface {
    Create(req CreateUserRequest) (*UserResponse, error)
    GetByID(id int) (*UserResponse, error)
}

// 4. 实现（具体业务逻辑）
type UserServiceImpl struct{}

func (s *UserServiceImpl) Create(req CreateUserRequest) (*UserResponse, error) {
    // 业务逻辑...
    return &UserResponse{ID: 1, Name: req.Name, Age: req.Age}, nil
}

func (s *UserServiceImpl) GetByID(id int) (*UserResponse, error) {
    // ...
    return nil, nil
}

// 5. HTTP Handler 使用接口
func createUserHandler(w http.ResponseWriter, r *http.Request) {
    var req CreateUserRequest
    json.NewDecoder(r.Body).Decode(&req)

    var svc UserService = &UserServiceImpl{}  // 依赖接口，而非具体实现
    user, err := svc.Create(req)
    // ...
}
```

> [!success] 这个模式的价值
> 用接口而非具体实现，好处是**可以随时替换实现**：
> - 测试时：传入 Mock 实现，不走数据库
> - 切换实现：换一个 UserServiceImpl2，不需要改 Handler 代码
> 这就是"**依赖注入**"，是后端工程化的基础。
