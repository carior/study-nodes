这是一个位标记（bit flag）系统，用于优化依赖追踪性能。解释如下：

## `w` 和 `n` 的含义

### 1. **类型定义**

`Dep` 是一个交叉类型，既是 `Set<ReactiveEffect>`，又包含 `TrackedMarkers`：

```3:19:packages/reactivity/src/dep.ts
export type Dep = Set<ReactiveEffect> & TrackedMarkers

/**
 * wasTracked and newTracked maintain the status for several levels of effect
 * tracking recursion. One bit per level is used to define whether the dependency
 * was/is tracked.
 */
type TrackedMarkers = {
  /**
   * wasTracked
   */
  w: number
  /**
   * newTracked
   */
  n: number
}
```

### 2. **两个标记位的含义**

- **`w` (wasTracked)**：上次 effect 运行时，该依赖是否已被追踪
- **`n` (newTracked)**：本次 effect 运行时，该依赖是否被追踪

### 3. **为什么在 Set 上设置这些属性？**

JavaScript 允许在对象上添加自定义属性。`Dep` 是一个 `Set` 实例，同时通过属性 `w` 和 `n` 存储追踪状态，这样可以：

- 复用 `Set` 的能力来管理 effects
- 在同一对象上存储追踪标记，避免额外查找

### 4. **位标记的工作原理**

Vue 3 使用位运算在单个数字中存储多个递归层级的标记：

```24:31:packages/reactivity/src/effect.ts
export let trackOpBit = 1

/**
 * The bitwise track markers support at most 30 levels of recursion.
 * This value is chosen to enable modern JS engines to use a SMI on all platforms.
 * When recursion depth is greater, fall back to using a full cleanup.
 */
const maxMarkerBits = 30
```

- `trackOpBit` 会根据递归深度动态变化：`trackOpBit = 1 << effectTrackDepth`
- 每一层使用不同的位，例如：第 1 层用 `1` (0b1)，第 2 层用 `2` (0b10)，第 3 层用 `4` (0b100)

### 5. **使用流程**

1. **初始化标记**（effect 开始运行时）：

```32:38:packages/reactivity/src/dep.ts
export const initDepMarkers = ({ deps }: ReactiveEffect) => {
  if (deps.length) {

}
```

2. **标记新追踪的依赖**（访问响应式属性时）：

```275:277:packages/reactivity/src/effect.ts

  dep.n |= trackOpBit // set newly tracked
  shouldTrack = !wasTracked(dep)
```

3. **清理未使用的依赖**（effect 运行结束时）：

```40:56:packages/reactivity/src/dep.ts
export const finalizeDepMarkers = (effect: ReactiveEffect) => {
  const { deps } = effect
  if (deps.length) {
    let ptr = 0
    for (let i = 0; i < deps.length; i++) {
      const dep = deps[i]
      if (wasTracked(dep) && !newTracked(dep)) {
        dep.delete(effect)
      } else {
        deps[ptr++] = dep
      }
      // clear bits
      dep.w &= ~trackOpBit
      dep.n &= ~trackOpBit
    }
    deps.length = ptr
  }
}
```

### 6. **优化目的**

通过比较 `w` 和 `n`：

- 如果 `wasTracked(dep)` 为 `true` 且 `newTracked(dep)` 为 `false`，说明该依赖不再被使用，可以安全移除
- 这避免了每次运行都清理所有依赖再重建，提高了性能

总结：这是 Vue 3 的依赖追踪优化机制，利用位标记高效地识别并清理不再使用的依赖，同时支持多层嵌套的 effect 递归调用。


这两行代码建立了双向的依赖关系。解释如下：

## 双向依赖关系的建立

### 代码含义

```292:295:packages/reactivity/src/effect.ts
if (shouldTrack) {
  // 建立起了 dep 和 activeEffect 的联系
  dep.add(activeEffect!)
  activeEffect!.deps.push(dep)
```

### 1. **`dep.add(activeEffect!)`** - 从 dep 到 effect 的引用

- 作用：将当前正在运行的 effect 添加到依赖集合 `dep` 中
- 含义：记录“哪些 effect 依赖了这个属性”
- 用途：当属性变化时，通过 `dep` 找到所有需要触发的 effect

### 2. **`activeEffect!.deps.push(dep)`** - 从 effect 到 dep 的引用

- 作用：将 `dep` 添加到 effect 的 `deps` 数组中
- 含义：记录“这个 effect 依赖了哪些属性”
- 用途：当 effect 停止时，可以清理所有相关的依赖关系

## 为什么需要双向引用？

### 场景 1：属性变化 → 触发 effect（需要 dep → effect）

当响应式属性变化时，需要通过 `dep` 找到所有依赖它的 effect：

```javascript

// 1. 从 targetMap 找到 state 的 depsMap
// 2. 从 depsMap 找到 'name' 对应的 dep
// 3. 遍历 dep 中的所有 effect 并执行
dep.forEach(effect => effect.run())
```

### 场景 2：effect 停止 → 清理依赖（需要 effect → dep）

当 effect 停止时，需要从所有相关的 `dep` 中移除这个 effect：

```144:152:packages/reactivity/src/effect.ts

  const { deps } = effect
  if (deps.length) {
    for (let i = 0; i < deps.length; i++) {
      deps[i].delete(effect)

    deps.length = 0
  }
}
```

## Effect 停止的场景

### 1. **手动调用 `stop()` 函数**


### 2. **EffectScope 停止时**

当 `EffectScope` 停止时，会停止其内部的所有 effect：

```81:107:packages/reactivity/src/effectScope.ts
stop(fromParent?: boolean) {
  if (this._active) {
    let i, l
    for (i = 0, l = this.effects.length; i < l; i++) {
      this.effects[i].stop()
    }
    for (i = 0, l = this.cleanups.length; i < l; i++) {
      this.cleanups[i]()
    }
    if (this.scopes) {
      for (i = 0, l = this.scopes.length; i < l; i++) {
        this.scopes[i].stop(true)
      }
    }
    // nested scope, dereference from parent to avoid memory leaks
    if (!this.detached && this.parent && !fromParent) {
      // optimized O(1) removal
      const last = this.parent.scopes!.pop()
      if (last && last !== this) {
        this.parent.scopes![this.index!] = last
        last.index = this.index!
      }
    }
    this.parent = undefined
    this._active = false
  }
}
```

**示例：**

```javascript
const scope = new EffectScope()

scope.run(() => {
  const state = reactive({ count: 0 })
  effect(() => {
    console.log(state.count)
  })
})

scope.stop()  // 停止 scope 时，内部的 effect 也会被停止
```

### 3. **Vue 组件卸载时**

在 Vue 3 中，每个组件实例都有一个 `EffectScope`，组件卸载时会停止该 scope，从而停止所有相关的 effect（如 `watch`、`computed`、`watchEffect` 等）。

### 4. **Effect 在运行过程中自己停止（deferStop 机制）**

特殊场景：effect 在执行过程中调用 `stop()` 停止自己：

```130:141:packages/reactivity/src/effect.ts
stop() {
  // stopped while running itself - defer the cleanup
  if (activeEffect === this) {
    this.deferStop = true
  } else if (this.active) {
    cleanupEffect(this)
    if (this.onStop) {
      this.onStop()
    }
    this.active = false
  }
}
```

**处理逻辑：**

- 如果 effect 正在运行（`activeEffect === this`），设置 `deferStop = true`
- 在 `run()` 方法的 `finally` 块中检查并执行停止：

## 清理依赖的过程

```144:152:packages/reactivity/src/effect.ts
function cleanupEffect(effect: ReactiveEffect) {
  const { deps } = effect
  if (deps.length) {
    for (let i = 0; i < deps.length; i++) {
      deps[i].delete(effect)
    }
    deps.length = 0
  }
}
```

- 遍历 effect 的 `deps` 数组
- 从每个 `dep`（Set）中删除该 effect
- 清空 `deps` 数组

## 总结

| 场景           | 触发方式                 | 使用场景                       |
| -------------- | ------------------------ | ------------------------------ |
| **手动停止**   | `stop(runner)`           | 需要手动控制 effect 的生命周期 |
| **Scope 停止** | `scope.stop()`           | 批量停止一组相关的 effect      |
| **组件卸载**   | Vue 自动调用             | 组件销毁时自动清理             |
| **自停止**     | effect 内部调用 `stop()` | 满足条件时自动停止             |

停止 effect 的主要目的是：

- 避免内存泄漏：清理不再需要的依赖关系
- 性能优化：停止不再需要的响应式追踪
- 生命周期管理：在合适的时机清理资源
