这是 TypeScript 的函数重载（Function Overloads）。

## 函数重载的作用

这种写法让同一个函数在不同参数下返回更精确的类型，同时共享同一个实现。

### 结构说明

```90:95:packages/reactivity/src/ref.ts
export function ref<T extends Ref>(value: T): T
export function ref<T>(value: T): Ref<UnwrapRef<T>>
export function ref<T = any>(): Ref<T | undefined>
export function ref(value?: unknown) {
  return createRef(value, false)
}
```

- 第 90-92 行：重载签名（Overload Signatures）
  - 仅用于类型检查，不参与运行时
  - 定义不同参数类型对应的返回类型
- 第 93-95 行：实现签名（Implementation Signature）
  - 实际的函数体，必须兼容所有重载签名

### 具体含义

1. `ref<T extends Ref>(value: T): T`
   - 传入已经是 `Ref` 的值时，直接返回该值（类型不变）

2. `ref<T>(value: T): Ref<UnwrapRef<T>>`
   - 传入普通值时，返回包装后的 `Ref`，并自动解包嵌套的 ref

3. `ref<T = any>(): Ref<T | undefined>`
   - 无参数调用时，返回 `Ref<T | undefined>`

### 使用示例

```typescript
// 情况1: 传入已经是 Ref 的值
const r1 = ref(someRef)  // 返回类型: Ref，不会再次包装

// 情况2: 传入普通值
const r2 = ref(42)       // 返回类型: Ref<number>

// 情况3: 不传参数
const r3 = ref()         // 返回类型: Ref<undefined>
```

### 为什么这样设计？

- 类型安全：根据输入给出精确的返回类型
- 代码复用：实现逻辑统一，避免重复代码
- 更好的 IDE 提示：编辑器能根据参数类型推断返回类型

这是 TypeScript 中常见的模式，在 Vue 3 源码中多处使用。

### 3. 然后设置 `name` 属性

- `get value()` 返回的是响应式 Proxy 对象
- 在这个 Proxy 对象上设置 `name` 会触发 Proxy 的 `set` handler（在 `baseHandlers.ts` 中）
- 不会触发 Ref 的 `set value()`，因为 `obj.value` 本身没有被重新赋值

## 对比两种情况

### 情况1：修改内部属性（你的代码）
```javascript

// 2. 在返回的 Proxy 对象上设置 name - 触发 Proxy 的 set handler
```

### 情况2：重新赋值整个 value
```javascript

// 触发 set value() - 因为 obj.value 本身被重新赋值了
```

## 总结

- `obj.value.name = '李四'` 需要先读取 `obj.value`，所以触发 `get value()`
- 返回的 `_value` 是响应式 Proxy，修改其属性会触发 Proxy 的 set handler
- 只有 `obj.value = ...` 这样的整体赋值才会触发 `set value()`

这是 JavaScript 的属性访问机制决定的：必须先读取对象引用，才能访问其属性。

4. effect 重新执行，再次访问 `computedObj.value`
5. `get value()` 检测到 `_dirty === true`，重新计算并更新缓存

### 5. 缓存机制（懒计算）

```58:68:packages/reactivity/src/computed.ts

```

- 懒计算：只在访问 `.value` 时计算
- 缓存：`_dirty === false` 时直接返回 `_value`
- 失效：依赖变化时 scheduler 将 `_dirty` 设为 `true`

### 6. 依赖关系图

```

    ↓ (依赖收集)
computed.effect (ReactiveEffect)
    ↓ (scheduler 触发)
computed._dirty = true
    ↓ (triggerRefValue)
computed.dep (包含使用 computed 的 effect)
    ↓ (触发)
effect(() => { ... computedObj.value ... })
```

### 7. 关键设计点

1. 双重依赖关系
   - computed 的 effect 依赖响应式数据（如 `obj.name`）
   - 使用 computed 的 effect 依赖 computed 本身

2. scheduler 机制
   - computed 的 effect 使用 scheduler，不立即执行
   - 依赖变化时只标记 `_dirty`，触发使用 computed 的 effect

3. 性能优化
   - 懒计算：不访问不计算
   - 缓存：`_dirty === false` 时复用结果
   - 按需更新：只在依赖变化时重新计算

### 总结

`computed` 通过 `_dirty` 标志实现懒计算和缓存：
- 首次访问或依赖变化后访问时计算
- 依赖未变化时直接返回缓存值
- 依赖变化时通过 scheduler 标记失效并触发依赖它的 effect

这样既保证了响应性，又避免了不必要的计算。

  dep: Dep,
  debuggerEventExtraInfo?: DebuggerEventExtraInfo
) {
  debugger
  let shouldTrack = false
  if (effectTrackDepth <= maxMarkerBits) {
    if (!newTracked(dep)) {
      dep.n |= trackOpBit // set newly tracked
      shouldTrack = !wasTracked(dep)
    }
  } else {
    // Full cleanup mode.
    shouldTrack = !dep.has(activeEffect!)

    activeEffect!.deps.push(dep)  // ⭐ 将 dep 添加到 activeEffect.deps 中

  age: 20
})

const computedObj = computed(() => {
  return obj.name + ' - ' + obj.age
})

// 外层 effect
effect(() => {
  const result = computedObj.value

  })

执行流程中的数据结构变化：

阶段1：创建 computed
```
computedObj.effect.deps = []  (空数组)
obj.name 的 dep = Set()  (空集合)
```

阶段2：外层 effect 执行，访问 computedObj.value
```
1. 外层 effect.run() 开始
   activeEffect = 外层effect

2. 访问 computedObj.value
   ├─> trackRefValue(computedObj)
   │    └─> computedObj.dep.add(外层effect)
   │    └─> 外层effect.deps.push(computedObj.dep)
   │
   └─> computed.effect.run() 开始
        activeEffect = computed.effect
        computed.effect.parent = 外层effect
        
        执行 getter: obj.name + obj.age
        ├─> 访问 obj.name
        │    └─> track(obj, 'name')
        │         └─> obj.name 的 dep.add(computed.effect)
        │         └─> computed.effect.deps.push(obj.name 的 dep)
        │
        └─> 访问 obj.age
             └─> track(obj, 'age')
                  └─> obj.age 的 dep.add(computed.effect)
                  └─> computed.effect.deps.push(obj.age 的 dep)
        
        computed.effect.run() 结束
        activeEffect = computed.effect.parent (外层effect)
```

阶段3：内层 effect 执行
```
内层 effect.run() 开始
  activeEffect = 内层effect
  内层effect.parent = 外层effect
  
  访问 obj.age
  └─> track(obj, 'age')
       └─> obj.age 的 dep.add(内层effect)
       └─> 内层effect.deps.push(obj.age 的 dep)
  
  内层 effect.run() 结束
  activeEffect = 内层effect.parent (外层effect)
```

最终数据结构：

```
obj.name 的 dep = Set {
  computed.effect  ← computed 依赖 name
}

obj.age 的 dep = Set {
  computed.effect,  ← computed 依赖 age
  内层effect        ← 内层 effect 依赖 age
}

computedObj.dep = Set {
  外层effect  ← 外层 effect 依赖 computed
}

外层effect.deps = [
  computedObj.dep
]

computed.effect.deps = [
  obj.name 的 dep,
  obj.age 的 dep
]

内层effect.deps = [
  obj.age 的 dep
]
```

当 obj.name 变化时：
1. 触发 `obj.name` 的 dep，执行 `computed.effect` 的 scheduler
2. scheduler 设置 `_dirty = true`，调用 `triggerRefValue(computedObj)`
3. 触发 `computedObj.dep`，执行 `外层effect`
4. `外层effect` 重新执行，再次访问 `computedObj.value`，重新计算

总结：
- 恢复 activeEffect 是为了维护嵌套 effect 的调用栈，确保依赖收集正确
- 数据结构通过双向引用（dep ↔ effect）建立依赖关系，支持精确的依赖追踪和触发

这就是 Vue 3 响应式系统的核心机制。

## 简单案例说明

假设有以下代码：

```javascript
// 1. 创建响应式对象
const obj = reactive({ name: '张三' })

// 2. 创建计算属性
const computedObj = computed(() => {
  return obj.name + ' 你好'  // computed 依赖 obj.name
})

// 3. 创建 effect，使用 computed
effect(() => {
  console.log(computedObj.value)  // effect 依赖 computedObj
})
```

### 数据结构关系

**1. computedObj (ComputedRefImpl)**
- `dep: Set<ReactiveEffect>`：存储依赖这个 computed 的 effect
  - 当 `effect` 读取 `computedObj.value` 时，`effect` 会被加入 `computedObj.dep`
- `effect: ReactiveEffect`：computed 自己的 effect（用于执行计算函数）
  - `deps: Dep[]`：存储这个 effect 依赖的所有 dep
    - 当 computed 的 getter 读取 `obj.name` 时，`obj.name` 的 dep 会被加入 `computedObj.effect.deps`

**2. effect (ReactiveEffect)**：使用 computed 的 effect
- `deps: Dep[]`：存储这个 effect 依赖的所有 dep
  - 当 effect 读取 `computedObj.value` 时，`computedObj.dep` 会被加入 `effect.deps`
- `fn: () => { ... }`：effect 的回调函数

### 依赖收集流程

1. 执行 `effect(() => { computedObj.value })`
   - `activeEffect = effect`
2. 读取 `computedObj.value`
   - `computedObj.dep.add(effect)` ← effect 被加入 computedObj.dep
   - `effect.deps.push(computedObj.dep)` ← computedObj.dep 被加入 effect.deps
3. computed 内部执行 getter：`() => obj.name + ' 你好'`
   - `activeEffect = computedObj.effect`
   - 读取 `obj.name`
   - `obj.name 的 dep.add(computedObj.effect)` ← computedObj.effect 被加入 obj.name 的 dep
   - `computedObj.effect.deps.push(obj.name 的 dep)` ← obj.name 的 dep 被加入 computedObj.effect.deps

### 触发更新流程

当 `obj.name = '李四'` 时：

1. 触发 `obj.name` 的 dep
   - 执行 `computedObj.effect`
2. computed 的 scheduler 执行
   - 设置 `computedObj._dirty = true`
   - 触发 `computedObj.dep`
3. 触发 `computedObj.dep` 中的所有 effect
   - 执行 `effect`
   - 重新读取 `computedObj.value`
   - computed 重新计算（因为 `_dirty = true`）

### 关键理解

- `computedObj.dep`：存储“谁依赖我”（哪些 effect 依赖这个 computed）
- `computedObj.effect.deps`：存储“我依赖谁”（computed 依赖哪些响应式数据）
- `effect.deps`：存储“我依赖谁”（effect 依赖哪些响应式数据/computed）

这样形成了双向依赖关系，确保数据变化时能正确触发更新。
